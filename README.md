# Agent-Driven Development Pipeline

Канонический подход к разработке с ИИ-агентами, который у нас стабильно работает в проде.

Суть в одной фразе: **один агент-оркестратор ставит задачу, запускает субагентов-исполнителей, а независимый субагент-ревьюер проверяет результат — человек только согласует контракт и принимает финальный диф.**

Этот гайд — про сам метод. Он не привязан к конкретному трекеру или языку: у нас Linear + в основном Python, у тебя может быть Jira и любой стек. Конкретная наша реализация (Python-оркестратор под Linear) лежит в [`reference/`](reference/) как живой пример, но повторять её один-в-один не нужно — важнее принципы ниже.

---

## Зачем это всё

Когда один агент и пишет код, и сам себя проверяет, и сам решает «готово» — он систематически переоценивает свою работу. Он не видит собственных дыр, ему «и так понятно», он срезает углы.

Подход чинит это тремя разделениями:

1. **Кто пишет ≠ кто проверяет.** Код пишет один субагент, проверяет — другой, с чистого листа, без доступа к самооценке первого. Независимый взгляд ловит то, что автор не замечает.
2. **Договорились о «готово» до кода, а не после.** Сначала контракт спринта (что именно делаем, что НЕ трогаем, как проверим), потом реализация. Это убирает расползание скоупа и «агент понял задачу не так».
3. **Задача самодостаточна.** Исполнителю не нужен устный контекст от человека. Вся неопределённость — продуктовая, контентная, юридическая — снимается ДО того, как задача уходит агенту (стандарт agent-ready, см. [`AGENT-READY-TICKETS.md`](AGENT-READY-TICKETS.md)).

## Роли

| Роль | Кто это | Что делает | Чего НЕ делает |
|------|---------|------------|----------------|
| **Оркестратор** | Главный агент (или человек + агент) | Выбирает задачу, доводит до agent-ready, согласует контракт, запускает исполнителей, собирает фидбэк, готовит финальный диф | Сам код обычно не пишет |
| **Исполнитель** | Субагент | Пишет код строго по контракту, отчитывается структурно | Не коммитит, не деплоит, не расширяет скоуп |
| **Ревьюер / оценщик** | Отдельный субагент | Проверяет работу с нуля, критично, не доверяя отчёту исполнителя | Не правит код сам — только выносит вердикт и список проблем |
| **Человек** | Ты | Согласует контракт, принимает финальный диф, жмёт «коммит/деплой» | Не пишет код руками и не пересылает контекст между агентами вручную |

Ключевой момент: ревьюер — это **отдельный** запуск субагента, которому НЕ дают самоотчёт исполнителя. Он судит по коду, а не по словам «я всё сделал».

## Поток работы (фазы)

```
1. ВЫБОР      Взять следующую задачу из трекера (Jira/Linear/…)
2. АНАЛИЗ     Разобрать требования + agent-ready preflight   ← ворота готовности
3. КОНТРАКТ   Согласовать «что значит готово» до кода
4. РЕАЛИЗАЦИЯ Запустить субагента-исполнителя ──────────┐
5. ПРОВЕРКА   Автоматические проверки (линтер, тесты)   │  до 3 раундов
6. ОЦЕНКА     Независимый ревьюер проверяет с нуля       │
7. ФИДБЭК     Вернуть найденное исполнителю на доработку ┘
8. ДИФ        Показать человеку чистый diff на одобрение
```

Большинство задач проходят за 1 раунд. 3 — потолок: если после третьего ревьюер всё ещё валит, задача останавливается и уходит человеку с полным списком, что не так.

### Фаза 1 — Выбор
Берёшь верхнюю задачу из бэклога (по приоритету, потом по дате). Ставишь статус «В работе». Показываешь человеку: что за задача, приоритет, описание — и спрашиваешь подтверждение перед запуском.

### Фаза 2 — Анализ + ворота готовности
Разбираешь требования: какие файлы, критерии приёмки, ограничения, зависимости. Затем **главная развилка** — классифицируешь задачу:

- `implementation-ready` — смысл понятен, можно кодить;
- `content/spec-missing` — не хватает самого контента / правил / спеки (текст страницы, бизнес-логика, UX-сценарии, контракт интеграции);
- `blocked` — нужно внешнее решение, доступ или подтверждение.

Если `content/spec-missing` — **стоп, код не запускаем.** Сначала дописываем в задачу сам содержательный артефакт (готовый текст, правила, сценарии), и только потом — скоуп и критерии. Нельзя просить исполнителя «придумать» продуктовый или юридический смысл. Подробно — в [`AGENT-READY-TICKETS.md`](AGENT-READY-TICKETS.md).

### Фаза 3 — Контракт спринта
До реализации фиксируешь договор:

```
КОНТРАКТ СПРИНТА — <ID задачи>

## Что делаем (Scope)
Конкретные изменения, не пересказ задачи:
1. <изменение> — какие файлы, какой подход
2. ...

## Что НЕ трогаем (Out of scope)
- <явное исключение, защита от расползания>

## Проверяемые критерии
| # | Критерий | Как проверяем | Ожидаемый результат |
|---|----------|---------------|---------------------|
| 1 | <текст>  | code review / запуск / API-вызов | <конкретное поведение> |

## Технический подход
- Паттерн (ссылка на существующий код, если есть)
- Ключевые решения (новый файл vs правка, выбор библиотеки)

## Риски
- <что может пойти не так и как подстрахуемся>
```

Прогоняешь контракт через субагента-ревьюера: покрывает ли он все критерии, реальны ли проверки, согласуется ли с кодовой базой, нет ли скрытых зависимостей, не перемудрили ли. Дорабатываешь — и контракт становится источником истины для фаз 4–6.

### Фаза 4 — Реализация
Запускаешь субагента-исполнителя. В промпт кладёшь: контракт (не сырую задачу), полное описание задачи для контекста, путь к проекту и его правила, релевантный контекст из памяти, и явные инструкции:

- читать исходники перед правками;
- реализовать **все** проверяемые критерии из контракта;
- держать стиль и паттерны проекта;
- **не коммитить** — только менять файлы;
- в конце вернуть структурный отчёт: список изменённых файлов, статус каждого критерия (DONE / NOT_DONE с причиной), замеченные риски.

### Фаза 5 — Проверка (автоматическая)
По изменённым файлам: файлы существуют, линтер чист, нет утёкших секретов (грепом по ключам/токенам/паролям), нет мусорных артефактов (`console.log`, `print(`, `debugger`, свежие `TODO/FIXME`), стиль не сломан. Критерии, проверяемые чтением кода, проверяешь **сам, независимо** — не веришь самоотчёту исполнителя.

### Фаза 6 — Независимая оценка
Отдельный субагент-ревьюер. Ему дают контракт и список изменённых файлов, но **не дают** самоотчёт исполнителя. Его инструкция жёсткая: «Ты не писал этот код. Оценивай критично, не будь снисходительным. Сломано — говори прямо». По каждому критерию: трассирует реализацию в коде, проверяет краевые случаи, обработку ошибок, регрессии. На выходе — вердикт PASS/FAIL по каждому критерию с доказательством (ссылка на код), общий вердикт, список критичных проблем.

Оценку можно пропустить только если задача тривиальна (правка конфига, доки, однострочник) И фаза 5 прошла без замечаний И в контракте нет критериев, требующих запуска.

### Фаза 7 — Цикл фидбэка
Если проверка или оценка нашли проблемы и раунд < 3 — собираешь структурный фидбэк и возвращаешь исполнителю:

```
РАУНД <N> — ФИДБЭК

## Блокеры (обязательно починить)
1. <проблема с путём к файлу и строкой>

## Предупреждения (желательно)
1. <...>

## Что уже работает (НЕ ломать)
<список прошедших критериев>
```

Последний блок критичен: исполнитель не должен сломать то, что уже работает, пока чинит остальное. Конкретика обязательна — «не работает» приводит к таким же расплывчатым правкам.

### Фаза 8 — Диф на одобрение
Готовишь чистый diff для человека: что за задача, что сделано, статус проверок, `diff --stat`, сам diff, предупреждения. И спрашиваешь: одобряешь — коммит / деплой / доработать? **Автодеплоя нет никогда** — финальное слово за человеком. После одобрения: коммит со ссылкой на ID задачи, статус «На ревью», комментарий с резюме, запись в память.

## Принципы (то, что держит качество)

- **Никогда не верь самооценке.** «DONE» исполнителя — подсказка, не доказательство. Всегда проверяй независимо.
- **Не пропускай независимого ревьюера на нетривиальных задачах.** Он ловит то, о чём исполнитель про свою же работу не расскажет.
- **Никогда не автодеплой.** Diff всегда идёт человеку на одобрение.
- **Одна задача за запуск.** Не пакетируй несколько задач в один прогон.
- **Не over-engineer.** Контракт задаёт скоуп — делай ровно его, без «заодно отрефакторю».
- **Фидбэк — конкретный.** Точные файлы и строки, а не «почини баг».
- **Не регрессируй на ретраях.** Каждый цикл фидбэка перечисляет, что уже работает.
- **Безопасность вперёд.** Если diff трогает аутентификацию, платежи или данные пользователей — отдельное внимание. Секрет в дифе — жёсткий стоп на любом раунде.

## Как привязать к своим инструментам

Метод не зависит от стека. Тебе нужно подставить три вещи:

**Трекер (у тебя Jira).** От трекера нужны всего четыре операции, без разницы как они называются: взять следующую задачу из бэклога, сменить статус (В работе → На ревью / Заблокировано), прочитать описание, оставить комментарий. В Jira это либо через MCP-интеграцию Atlassian, либо через REST API, либо руками. Маппинг статусов: `Todo / In Progress / Review / Blocked` → твои названия колонок.

**Язык(и).** Подход языко-нейтральный. Всё специфичное для стека живёт в одном месте — в Definition of Done задачи: туда подставляешь свои команды тестов и линтера (`go test ...`, `npm test`, `pytest`, `cargo test` — что угодно). Исполнитель просто выполняет то, что написано в DoD.

**Запуск.** Два варианта:
- **Вручную в чате** — самый простой старт, скриптов не надо. В одном диалоге просишь агента быть оркестратором: даёшь задачу, просишь составить контракт, потом запускать субагентов на реализацию и отдельно на ревью (в Claude Code / Cursor это инструмент `Task` / субагенты). Всё описанное выше делается «руками» оркестратора.
- **Скриптом-оркестратором** — когда поток устоялся, автоматизируешь его, чтобы не пересылать контекст между агентами вручную. Наш пример — [`reference/orchestrate.py`](reference/orchestrate.py): он сам берёт задачу из трекера, заводит git worktree, гоняет субагентов до 3 раундов, ставит статус и пишет комментарий. Это образец логики, а не готовый продукт под Jira — но из него видно, как фазы ложатся в код.

## Минимальный старт за 10 минут

1. Возьми одну небольшую задачу из своего Jira.
2. Доведи её до agent-ready по чеклисту из [`AGENT-READY-TICKETS.md`](AGENT-READY-TICKETS.md) — это половина успеха.
3. В чате с агентом-оркестратором: «составь контракт спринта по этой задаче».
4. Согласуй контракт.
5. «Запусти субагента-исполнителя по контракту, не коммить».
6. «Запусти отдельного субагента-ревьюера, дай ему контракт и список изменённых файлов, но НЕ давай отчёт исполнителя».
7. Прогони цикл фидбэка, если ревьюер нашёл проблемы.
8. Посмотри финальный diff сам, реши — коммитить или дорабатывать.

После пары прогонов станет видно, что стоит автоматизировать — тогда и подсмотри в [`reference/`](reference/).
