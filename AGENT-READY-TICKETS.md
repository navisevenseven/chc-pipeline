# Agent-Ready задачи — стандарт постановки

Задача считается **agent-ready**, если её можно отдать агенту-исполнителю без устного контекста от человека. Это вторая половина подхода (первая — [`README.md`](README.md)): хороший поток с оркестратором и ревьюером бесполезен, если на вход приходит задача «поправь матчинг по сценарию 6» — коллеге понятно, агенту нет.

Стандарт трекер-нейтральный: примеры даны абстрактно, подставляй свой Jira.

---

## Главный инвариант

**Agent-ready задача не перекладывает продуктовую, контентную или юридическую неопределённость на исполнителя.**

Если задаче нужен текст страницы, бизнес-правила, legal/privacy-текст, UX-copy, сценарии или edge cases — в задаче должен лежать **сам содержательный артефакт**, а не инструкция «агент должен написать/решить/придумать».

✅ Правильно:
```markdown
## Готовый контент страницы
<финальный или почти финальный текст, который надо реализовать>
```

❌ Неправильно:
```markdown
## Заметки для агента
Агент должен написать консервативный текст Privacy Policy.
```

Агент отлично пишет код по спецификации, но не должен в одиночку решать, что значит твой продукт, что разрешает юрист и какой тон у бренда. Эти решения — на человеке, до постановки задачи.

---

## Ворота готовности (preflight)

Перед тем как оформлять задачу по шаблону, классифицируй её:

- `implementation-ready` — смысл определён, можно упаковывать в scope / критерии / DoD;
- `content/spec-missing` — сначала надо написать сам контент / спеку / правила;
- `blocked` — не хватает внешнего решения, доступа или подтверждения.

Если `content/spec-missing` — нельзя сразу писать только критерии приёмки. Сначала добавь в задачу источник истины:

- для публичных страниц / Privacy / Support / Landing — `## Готовый контент страницы`;
- для бизнес-логики — `## Правила` + примеры;
- для UX-потоков — `## Сценарии` + ожидаемое поведение;
- для интеграций — точный контракт: payloads, ошибки, edge cases.

И только потом — `Файлы в scope`, `Критерии приёмки`, `DoD`, `План проверки`.

## Обязательная структура

Каждая задача содержит секции **в этом порядке**, ни одна обязательная не пустая:

```markdown
## Контекст
Зачем задача, какую боль решает, как сейчас работает код (ссылки на файлы/коммиты/прошлые задачи). 3–10 строк.

## Файлы в scope
Конкретные пути:
- `src/matching/service.<ext>`
- `migrations/0007_pricing_snapshot.up.sql`
Новый файл — указать путь и назначение.

## Вне scope
Что НЕ трогать. Защита от расползания:
- Не менять публичный API (только внутреннее)
- Не рефакторить соседний модуль

## Критерии приёмки (AC)
- [ ] AC1: <что проверяется> — проверка: <как: команда / вывод / endpoint>
- [ ] AC2: ...
Каждый AC имеет конкретную проверку. «Работает» — не проверка.

## Готовность (DoD)
- [ ] Код реализован
- [ ] Тесты добавлены/прошли: `<конкретная команда>`
- [ ] Линтер чист: `<конкретная команда>`
- [ ] Коммит: `<type>: <desc> (<ID задачи>)`
- [ ] Push в ветку `<branch>`
- [ ] (если применимо) PR открыт: base=`<base>`, head=`<branch>`
- [ ] Комментарий в задаче: summary + diff stat + ссылка на коммит/PR

## План проверки
Пошагово, как убедиться что задача реально решена: env setup, команды, ожидаемый вывод.

## Зависимости
- <ID> — должна быть Done до этой
- Внешние: <API / сервис / другая команда>
- Если нет — написать "None"

## Блокирующий статус (опционально)
Только если задачу нельзя брать сейчас. С явной причиной:
- "Ждём, пока <ID> протестируют в проде (~1 неделя)"
- "Нужен ручной дамп данных от DBA"

## Заметки для агента
Очевидное человеку, но не агенту:
- "Миграции применяются автоматически при деплое, руками не запускать"
- "Секреты в <путь>, в репозиторий не коммитим"
- "На проекте pgx v5, не v4"
- "Тестам нужен локальный сервис на :3000 — поднять через docker compose"
```

## Самопроверка перед отправкой

Любой Hard failure → доработай или задай уточняющий вопрос. Если вопросов несколько — **задавай их одним списком**, не по одному.

### Hard failures (блокируют)

| # | Правило |
|---|---------|
| R1 | Секции `Контекст`, `Файлы в scope`, `Критерии приёмки`, `DoD`, `План проверки`, `Зависимости` — присутствуют и непусты |
| R2 | У каждого AC есть `проверка:` |
| R3 | В DoD указана конкретная команда тестов (если тесты применимы) |
| R4 | В DoD указана ветка для push (соглашение: `feature/<id>-<short-slug>`) |
| R5 | Нет плейсхолдеров: `TBD`, `TODO`, `to be defined`, «подробности уточним» |
| R6 | Нет противоречий между `Критерии приёмки` и `Вне scope` |
| R7 | Для content-bearing задач есть `## Готовый контент страницы` / `## Правила` / `## Сценарии` или другой явный источник истины; нельзя ограничиться «агент должен написать/придумать» |

### Soft warnings (желательно)

| # | Правило |
|---|---------|
| W1 | `Контекст` содержит хотя бы одну ссылку (файл/коммит/задача) |
| W2 | Хотя бы 2 AC (задача не монолитная) |
| W3 | `Заметки для агента` есть, если у проекта своя специфика |

---

## Анти-паттерны (почему задача падает)

- **«Поправь баг в X»** — нет AC, нет проверки. FAIL R2.
- **«Обнови зависимости»** — нет scope, нет `Вне scope`. FAIL R1.
- **«Как обсуждали в чате»** — нет `Контекста` со ссылками. FAIL W1, при полном отсутствии контекста — R1.
- **AC: «работает быстро»** — нет проверки. FAIL R2. Правильно: `AC: P99 latency <= 200ms — проверка: wrk -t4 -c100 ...`.
- **DoD без команды тестов, когда они есть** — FAIL R3.

---

## Почему именно так

Стандарт делает ровно одно: переносит всю неопределённость на этап постановки, где её снимает человек с контекстом, а не агент в процессе кодинга. Чем строже задача на входе, тем меньше раундов фидбэка на выходе — и тем реже независимый ревьюер из [`README.md`](README.md) валит результат. Две практики работают в паре: agent-ready задача на входе + независимая проверка на выходе.
