# reference/ — наша конкретная реализация (пример)

Это **не** то, что нужно копировать дословно. Это образец того, как фазы из [`../README.md`](../README.md) ложатся в реальный код — у нас под Linear и в основном на Python.

У тебя Jira и другой стек, так что этот скрипт напрямую не запустится. Смотри на него как на иллюстрацию логики: как берётся задача из трекера, как заводится отдельный git worktree на задачу, как гоняются субагенты до 3 раундов, как ставится статус и пишется комментарий.

| Файл | Что это |
|------|---------|
| `orchestrate.py` | Оркестратор целиком, только стандартная библиотека Python. Берёт задачу из Linear → agent-ready preflight → контракт → `claude -p` в worktree → проверки → статус Review + комментарий |
| `projects.json` | Реестр проектов: UUID команд и статусов трекера, имена env-переменных с путями к репозиториям |
| `chc-pipeline` | Bash-обёртка над `orchestrate.py` |

## Что здесь стоит подсмотреть

- **Один worktree на задачу** — каждая задача реализуется в изолированном git worktree рядом с репозиторием, чтобы не мешать основной ветке.
- **До 3 раундов** — цикл implement → validate → evaluate → feedback с жёстким потолком.
- **Контракт как источник истины** — субагент получает контракт, а не сырую задачу.
- **Артефакты на диск** — контракт, промпты, ответы агента и отчёты валидации пишутся в `.chc-pipeline/artifacts/<ID>/`, чтобы прогон был прозрачным и воспроизводимым.

## Чтобы перенести на свой стек

1. Замени операции трекера (в коде — вызовы Linear API) на Jira: взять задачу, сменить статус, прочитать, прокомментировать.
2. Подставь свои команды тестов и линтера — они и так задаются через DoD задачи и env-переменную с тест-командой.
3. Движок исполнителя (`claude -p`) оставь или замени на свой агент-CLI.
